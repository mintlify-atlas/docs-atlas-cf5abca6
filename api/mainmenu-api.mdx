description: "Complete reference for the Luanti main menu Lua API, including callbacks, game management, server list, UI helpers, filesystem, HTTP, and async operations."
---

The main menu in Luanti is defined as a formspec by Lua scripts located in `builtin/mainmenu/`. You can extend it by understanding the API described here.

<Note>
Formspec language syntax is described in `lua_api.md`. When embedding image or 3D model paths in a formspec, sanitize them first with `core.formspec_escape(path)` — paths in subdirectories use different delimiters on different operating systems.
</Note>

---

## Callbacks

The main menu exposes three top-level callbacks. Define them to respond to user actions.

### `core.button_handler`

```lua
function core.button_handler(fields)
    -- fields = {name1 = value1, name2 = value2, ...}
end
```

Called whenever a button is pressed in the main menu formspec.

<ParamField body="fields" type="table">
  A table of field name/value pairs from the submitted form.
</ParamField>

```lua
function core.button_handler(fields)
    if fields.btn_singleplayer then
        -- user clicked the singleplayer button
    end
end
```

---

### `core.event_handler`

```lua
function core.event_handler(event)
end
```

Called when a menu-level event is emitted.

<ParamField body="event" type="string">
  The event name. Standard values:
  - `"MenuQuit"` — user quit the menu (derived from `quit`)
  - `"FullscreenChange"` — window fullscreen state changed
  - `"Refresh"` — server list refresh (custom event)
</ParamField>

---

### `core.on_before_close`

```lua
function core.on_before_close()
end
```

Called immediately before the menu closes, either to exit the engine or to join a game. Use this to save state.

## Starting and Closing

### `core.start`

```lua
core.start()
```

Starts a game session using the `gamedata` table. Populate `gamedata` before calling this:

```lua
gamedata = {
    playername     = "player",
    password       = "",
    address        = "luanti.example.org",
    port           = 30000,
    selected_world = 1,   -- 0 for client (CSM-only) mode
    singleplayer   = false,
}
core.start()
```

<ResponseField name="playername" type="string">
  The player's username.
</ResponseField>

<ResponseField name="password" type="string">
  The player's password.
</ResponseField>

<ResponseField name="address" type="string">
  Server IP or domain name. Empty string for a local server.
</ResponseField>

<ResponseField name="port" type="number">
  Server port number.
</ResponseField>

<ResponseField name="selected_world" type="number">
  Index into the worlds list. Use `0` for client mode.
</ResponseField>

<ResponseField name="singleplayer" type="boolean">
  Whether this is a singleplayer session.
</ResponseField>

---

### `core.close`

```lua
core.close()
```

Exits the Luanti engine entirely.

## Games and Content

### `core.get_games`

```lua
core.get_games()
```

Returns a table of all installed games (can be called in async context):

```lua
{
    {
        id             = "minetest",
        path           = "/usr/share/minetest/games/minetest",
        gamemods_path  = "/usr/share/minetest/games/minetest/mods",
        title          = "Minetest Game",
        menuicon_path  = "/usr/share/minetest/games/minetest/menu/icon.png",
        author         = "celeron55",
        aliases        = {},
    },
    ...
}
```

<Note>
The `name` field is deprecated. Use `title` instead.
</Note>

---

### `core.get_content_info`

```lua
core.get_content_info(path)
```

Returns metadata about any installed content (mod, modpack, game, or texture pack) at `path`:

```lua
{
    name             = "technical_id",
    type             = "mod",  -- or "modpack", "game", "txp", "unknown"
    title            = "Human Readable Title",
    description      = "description text",
    author           = "author",
    path             = "path/to/content",
    textdomain       = "textdomain",
    depends          = {"mod", "names"},  -- mods only
    optional_depends = {"mod", "names"},  -- mods only
}
```

---

### `core.get_mod_list`

```lua
core.get_mod_list(path, virtual_path)
```

Returns a flat list of mod and modpack information found within the specified path. Each entry contains: `name`, `author`, `release`, `description`, `path`, `virtual_path`, `is_name_explicit`, `is_modpack`, `modpack_depth`.

---

### `core.check_mod_configuration`

```lua
core.check_mod_configuration(world_path, mod_paths)
```

Validates that a set of mods can be loaded together for a given world.

<ParamField body="world_path" type="string">
  Absolute path to the world directory.
</ParamField>

<ParamField body="mod_paths" type="table">
  List of enabled mod paths.
</ParamField>

Returns:

```lua
{
    is_consistent  = true,
    unsatisfied_mods = {},   -- list of mod specs with unmet dependencies
    satisfied_mods   = {},   -- list of mod specs that resolved correctly
    error_message    = "",   -- error string, or nil
}
```

---

### `core.get_content_translation`

```lua
core.get_content_translation(path, domain, string)
```

Translates `string` using the specified `domain` in the content directory at `path`.

```lua
local translated = core.get_content_translation(
    "mods/mymod",
    "mymod",
    core.translate("mymod", "Hello World")
)
```

## HTTP Requests

<Warning>
HTTP functions are only available if Luanti was built with cURL support.
</Warning>

### `core.get_http_api`

```lua
local http = core.get_http_api()
```

Returns an `HTTPApiTable` with three functions:

---

### `HTTPApiTable.fetch_sync`

```lua
local result = http.fetch_sync(request)
```

Performs a synchronous HTTP request. Blocks until complete.

---

### `HTTPApiTable.fetch_async`

```lua
local handle = http.fetch_async(request)
```

Starts an asynchronous HTTP request. Returns a handle.

---

### `HTTPApiTable.fetch_async_get`

```lua
local result = http.fetch_async_get(handle)
```

Checks the status of an async request. Call repeatedly until `result.completed` is `true`.

---

### HTTPRequest definition

```lua
{
    url          = "https://example.org/api",
    timeout      = 10,       -- connection timeout in seconds (default: 3)
    post_data    = "raw body" -- or {field1="val", field2="val"}
                             -- omit for a GET request
    user_agent   = "MyMod/1.0",       -- optional
    extra_headers = { "Accept: application/json" },  -- optional
    multipart    = false,    -- optional, default false
}
```

### HTTPRequestResult definition

```lua
{
    completed = true,    -- request is finished
    succeeded = true,    -- request was successful
    timeout   = false,   -- request timed out
    code      = 200,     -- HTTP status code
    data      = "...",   -- response body
}
```

## Async Operations

### `core.handle_async`

```lua
core.handle_async(async_job, parameters, finished)
```

Executes a function asynchronously on a worker thread.

<ParamField body="async_job" type="function">
  A function that takes one parameter and returns one parameter. Runs off the main thread.
</ParamField>

<ParamField body="parameters" type="table">
  The parameter table passed to `async_job`.
</ParamField>

<ParamField body="finished" type="function">
  Callback called on the main thread once `async_job` completes. Receives the return value of `async_job`.
</ParamField>

```lua
core.handle_async(
    function(params)
        -- runs on worker thread
        return { result = params.x * 2 }
    end,
    { x = 21 },
    function(result)
        core.log("Result: " .. tostring(result.result))  -- 42
    end
)
```

<Warning>
Async jobs have **no access to global Lua variables** and only a limited set of functions. Functions that modify the menu (like `core.start`, `core.close`, `core.show_path_select_dialog`) are unavailable inside `async_job`.
</Warning>

## Background Music

The main menu plays background music automatically. It looks for a sound named `main_menu` in `$USER_PATH/sounds/`, using the same naming conventions as regular sounds (the engine picks randomly from `main_menu.ogg`, `main_menu.0.ogg`, etc.). It plays looped at gain `1.0`.
